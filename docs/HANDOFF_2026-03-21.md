## 1. Что это за проект сейчас

Проект уже не является просто bootstrap-заготовкой. На текущем этапе это рабочий single-node control plane для `Xray-core + VLESS + REALITY`, состоящий из:

- `api`: NestJS + TypeScript + Prisma + PostgreSQL
- `web`: React + TypeScript + Vite
- `xray`: data plane на `443/tcp`
- `caddy`: reverse proxy панели на `8443/tcp`
- host-level `ufw` и `fail2ban`

Архитектурная идея:

- `443` зарезервирован под VPN-трафик Xray
- панель живет на `8443`, потому что домен под публичный TLS для панели пока не задан
- `Xray` и `api` связаны через внутреннюю control network
- PostgreSQL является каноническим control-plane хранилищем
- конфиг Xray не редактируется руками на проде, а рендерится из шаблона

## 12. Update: March 26, 2026

### Что сделано сегодня

За текущую сессию был закрыт следующий блок работ:

- реализован safe `restore` preflight flow без destructive запуска из UI:

  - в UI backups появился отдельный restore plan с явными командами `--dry-run` и `--yes-restore`, статусом preflight и предупреждениями
  - host-side `infra/scripts/restore.sh` получил `--dry-run`, валидацию архива, защиту по абсолютному пути и более безопасную логику работы с runtime config Xray
- разобраны замечания code review из shared-chat и прогнаны серверные проверки:
  - typecheck-ошибки по Prisma оказались следствием отсутствующего `prisma generate`
  - один API test падал из-за отсутствующего `TOTP_ENCRYPTION_SECRET` в test env
  - два web test были реально сломаны и исправлены
  - после этого в чистом containerized workspace стали зелёными `typecheck`, `test`, `build`
- в процессе деплоя всплыл production bug в Docker build API:
  - runtime image стартовал без `apps/api/dist/main.js`
  - причина: stale `*.tsbuildinfo` в Docker context и инкрементальная сборка Nest/TypeScript
  - исправлено через исключение `*.tsbuildinfo` из `.dockerignore`, очистку build-info перед `npm run build -w apps/api` и явную проверку `test -f /app/apps/api/dist/main.js` прямо в Dockerfile
- после hotfix прод снова поднят и проверен
- доведён runtime enforcement для `deviceLimit` / `ipLimit`:
  - `XrayService` теперь при usage snapshot читает online-IP telemetry из Xray Stats API
  - `activeConnections` в текущем MVP теперь отражает число одновременных online endpoint, а не просто `0/1`
  - при превышении лимитов клиент автоматически переводится в `BLOCKED`, удаляется из live Xray runtime и получает audit trail с причиной
  - для совместимости добавлен fallback с `GetStatsOnlineIpList` на `GetStatsOnline`
  - в help-разделе добавлена явная подсказка, как именно эти лимиты трактуются в текущем runtime
- исправлен раздражающий browser `Basic Auth` popup на панели:
  - причиной был запрос браузера к `/favicon.ico`, который раньше тоже попадал под `basic_auth` и мог повторно вызывать challenge
  - в Caddy вынесены `favicon/manifest` asset paths из `basic_auth`

## Update 2026-03-26: Automatic backup policy

### Что изменили

- Добавлен встроенный scheduler в `BackupsService`: API теперь сам периодически проверяет backup policy и при необходимости создаёт новый backup без ручного действия из UI.
- Политика вынесена в env:
  - `BACKUP_AUTO_CREATE_ENABLED=true`
  - `BACKUP_AUTO_CREATE_INTERVAL_DAYS=5`
  - `BACKUP_AUTO_MAINTENANCE_INTERVAL_MS=3600000`
  - `BACKUP_RETENTION_DAYS=14`
- Автобэкап использует тот же безопасный pipeline, что и ручной backup:
  - общий `createSnapshot(...)`
  - тот же tar/archive flow
  - тот же checksum / manifest
  - тот же `pruneExpiredBackups()`
- В `GET /api/backups` policy теперь явно возвращает:
  - `autoCreateEnabled`
  - `autoCreateIntervalDays`
  - `retentionDays`
- В вебе на странице backups добавлено отображение policy: видно, что автобэкап включён и создаётся каждые 5 дней.

### Как это работает

- При старте API через ~15 секунд выполняется bootstrap-check policy.
- Дальше scheduler крутится раз в час и решает, пора ли создавать новый архив.
- Новый backup создаётся только если:
  - автобэкап не выключен
  - нет свежего `CREATING` snapshot, чтобы не словить дубль во время гонки
  - с момента последнего успешного backup прошло не меньше `5` дней
- Если backup ещё не пора создавать, maintenance всё равно делает cleanup по retention policy.
- Cleanup остаётся безопасным: старые архивы старше `14` дней помечаются как `PRUNED` и удаляются с диска через уже существующий prune flow.

### Что важно по безопасности

- Existing backup/restore flow не менялся в destructive сторону:
  - restore-команды прежние
  - формат архива прежний
  - старые архивы и конфиги не мигрировались и не переписывались
- Ретеншн применяется по age, а не по fixed-count. То есть удаляется не просто "самый старый", а любой backup старше 14 дней.
- На момент rollout последний успешный backup был `2026-03-25`, поэтому scheduler не должен был создавать дубликат сразу после deploy.

- То есть:
  - `Xray-core` здесь выступает runtime-движком
  - `VLESS` это протокол клиента/inbound
  - `REALITY` это transport security / camouflage layer поверх этого VLESS-контура

- Корректное краткое описание текущего продукта:
  - control plane для `Xray-core`
  - single-VPS deployment
