## 4. Актуальная схема deploy

Сейчас есть два режима:

- `PANEL_TLS_MODE=ip`
  - `Xray` слушает `443`
  - panel доступна на `https://IP:8443`
- `PANEL_TLS_MODE=domain`
  - `HAProxy` принимает `80/443`
  - panel domain по SNI идёт в `Caddy`
  - остальной TLS на `443` уходит в `Xray`
  - legacy `8443` сохраняется как fallback

## 5. Что важно помнить по эксплуатации

- На VPS обычный `git fetch/pull` из private GitHub сейчас не рабочий по HTTPS без отдельной авторизации.
- Поэтому фактические обновления на сервер попадали через manual sync / bundle flow.
- Из-за этого на сервере `HEAD` актуальный, но локальный `origin/main` на VPS может быть устаревшим и не отражать реальное состояние checkout.

Практический вывод:

- состояние прод-сервера проверяем по `git rev-parse HEAD`, а не по `git status ... origin/main`;
- перед следующими выкладками лучше либо вернуть рабочий SSH/deploy-key flow, либо осознанно продолжать bundle-based deploy.

## 7. Важный архитектурный нюанс

Для текущего стека `VLESS + REALITY` нужно помнить:

- это proxy/control-plane схема, а не классический L3 VPN с уникальным внутренним IP на каждого клиента;
- `deviceLimit` и `ipLimit` у нас считаются по online endpoint / source IP telemetry из Xray;
- если понадобится именно уникальная внутренняя адресация для каждого клиента, это уже отдельная транспортная схема, а не текущий `VLESS/REALITY` runtime.
